When working with non-deterministic systems that make model-based decisions (e.g., agents powered by LLMs), it can be useful to examine their decision-making process in detail:

1. <Icon icon="bulb" size={16} /> **Understand reasoning**: Analyze the steps that led to a successful result.
2. <Icon icon="bug" size={16} /> **Debug mistakes**: Identify where and why errors occurred.
3. <Icon icon="search" size={16} /> **Explore alternatives**: Test different paths to uncover better solutions.

LangGraph supports time travel through [checkpoints](/oss/langgraph/persistence#checkpoints). Both replay and fork work by resuming from a prior checkpoint. In both cases:

- **Nodes before the checkpoint** are not re-executed. Their results are already saved in the checkpoint.
- **Nodes after the checkpoint** re-execute. Code runs again, including any LLM calls or API requests (which may produce different results).

The difference is whether you call @[`update_state`] before resuming:

- **[Replay](#replay-a-past-execution)**: Invoke directly from a prior checkpoint. [Interrupt](/oss/langgraph/interrupts) resume values are cached and reused automatically.
- **[Fork](#fork-and-explore-alternatives)**: Call `update_state` on a prior checkpoint first, then invoke. Cached interrupt resume values are cleared — interrupts are re-triggered and wait for new input. You can fork [with modified state](#with-state-updates) or [without](#without-state-updates).

| | **Replay** | **Fork** |
|---|---|---|
| **How** | Invoke with a prior `checkpoint_id` | Call `update_state` on a prior checkpoint, then invoke |
| **State** | Original checkpoint state | Modified ([with updates](#with-state-updates)) or unchanged ([without](#without-state-updates)) |
| **Interrupts** | Use cached resume values — NOT re-triggered | Cached values cleared — interrupts ARE re-triggered |

<Warning>

`update_state` does **not** roll back a thread. It creates a new checkpoint that branches from the specified point. The original execution history remains intact.

</Warning>

## Replay a past execution

Invoke the graph with a prior checkpoint's config to replay from that point. Nodes after the checkpoint re-execute, and [interrupt](/oss/langgraph/interrupts) resume values are cached and reused automatically.

![Replay](/oss/images/re_play.png)

:::python
Use @[`get_state_history`] to find the checkpoint you want to replay from, then call @[`invoke`][CompiledStateGraph.invoke] with that checkpoint's config:

```python
from langgraph.graph import StateGraph, START
from langgraph.checkpoint.memory import InMemorySaver
from typing_extensions import TypedDict, NotRequired
import uuid

class State(TypedDict):
    topic: NotRequired[str]
    joke: NotRequired[str]


def generate_topic(state: State):
    return {"topic": "socks in the dryer"}


def write_joke(state: State):
    return {"joke": f"Why do {state['topic']} disappear? They elope!"}


checkpointer = InMemorySaver()
graph = (
    StateGraph(State)
    .add_node("generate_topic", generate_topic)
    .add_node("write_joke", write_joke)
    .add_edge(START, "generate_topic")
    .add_edge("generate_topic", "write_joke")
    .compile(checkpointer=checkpointer)
)

# Step 1: Run the graph
config = {"configurable": {"thread_id": str(uuid.uuid4())}}
result = graph.invoke({}, config)

# Step 2: Find a checkpoint to replay from
history = list(graph.get_state_history(config))
# History is in reverse chronological order
for state in history:
    print(f"next={state.next}, checkpoint_id={state.config['configurable']['checkpoint_id']}")

# Step 3: Replay from a specific checkpoint
# Find the checkpoint before write_joke
before_joke = next(s for s in history if s.next == ("write_joke",))
replay_result = graph.invoke(None, before_joke.config)
# write_joke re-executes (runs again), generate_topic does not
```
:::

:::js
Use @[`getStateHistory`] to find the checkpoint you want to replay from, then call @[`invoke`][CompiledStateGraph.invoke] with that checkpoint's config:

```typescript
import { v4 as uuidv4 } from "uuid";
import { StateGraph, MemorySaver, START } from "@langchain/langgraph";

const StateAnnotation = Annotation.Root({
  topic: Annotation<string>(),
  joke: Annotation<string>(),
});

function generateTopic(state: typeof StateAnnotation.State) {
  return { topic: "socks in the dryer" };
}

function writeJoke(state: typeof StateAnnotation.State) {
  return { joke: `Why do ${state.topic} disappear? They elope!` };
}

const checkpointer = new MemorySaver();
const graph = new StateGraph(StateAnnotation)
  .addNode("generateTopic", generateTopic)
  .addNode("writeJoke", writeJoke)
  .addEdge(START, "generateTopic")
  .addEdge("generateTopic", "writeJoke")
  .compile({ checkpointer });

// Step 1: Run the graph
const config = { configurable: { thread_id: uuidv4() } };
const result = await graph.invoke({}, config);

// Step 2: Find a checkpoint to replay from
const states = [];
for await (const state of graph.getStateHistory(config)) {
  states.push(state);
}

// Step 3: Replay from a specific checkpoint
const beforeJoke = states.find((s) => s.next.includes("writeJoke"));
const replayResult = await graph.invoke(null, beforeJoke.config);
// writeJoke re-executes (runs again), generateTopic does not
```
:::

## Fork and explore alternatives

Fork creates a new branch from a past checkpoint. Call @[`update_state`] on a prior checkpoint to create the fork, then @[`invoke`][CompiledStateGraph.invoke] with `None` to continue execution.

![Fork](/oss/images/checkpoints_full_story.jpg)

### With state updates

Use this to change something in a past execution and see what happens — for example, to correct a wrong tool call, try a different LLM response, or test multiple branches from the same point.

:::python
```python
# Find checkpoint before write_joke
history = list(graph.get_state_history(config))
before_joke = next(s for s in history if s.next == ("write_joke",))

# Fork: update state to change the topic
fork_config = graph.update_state(
    before_joke.config,
    values={"topic": "chickens"},
)

# Resume from the fork — write_joke re-executes with the new topic
fork_result = graph.invoke(None, fork_config)
print(fork_result["joke"])  # A joke about chickens, not socks
```
:::

:::js
```typescript
// Find checkpoint before writeJoke
const states = [];
for await (const state of graph.getStateHistory(config)) {
  states.push(state);
}
const beforeJoke = states.find((s) => s.next.includes("writeJoke"));

// Fork: update state to change the topic
const forkConfig = await graph.updateState(
  beforeJoke.config,
  { topic: "chickens" },
);

// Resume from the fork — writeJoke re-executes with the new topic
const forkResult = await graph.invoke(null, forkConfig);
console.log(forkResult.joke); // A joke about chickens, not socks
```
:::

### Without state updates

Use `as_node="__copy__"` to fork without modifying state. This creates a new branch point from the checkpoint, clearing cached [interrupt](/oss/langgraph/interrupts) resume values so interrupts are re-triggered.

:::python
```python
# Fork without changing state — just re-trigger interrupts
fork_config = graph.update_state(
    before_ask.config,
    values=None,
    as_node="__copy__",
)

result = graph.invoke(None, fork_config)
# Interrupts are re-triggered, waiting for new Command(resume=...)
```
:::

:::js
```typescript
// Fork without changing state — just re-trigger interrupts
const forkConfig = await graph.updateState(
  beforeAsk.config,
  null,
  { asNode: "__copy__" },
);

const result = await graph.invoke(null, forkConfig);
// Interrupts are re-triggered, waiting for new Command({ resume: ... })
```
:::

### Control which node runs next with `as_node`

When you call @[`update_state`], the values are applied using the specified node's writers (including [reducers](/oss/langgraph/graph-api#reducers)). The checkpoint records that node as having produced the update, and execution resumes from that node's successors in the graph.

By default, LangGraph infers `as_node` from the checkpoint's version history — whichever node last updated state. When forking from a specific checkpoint, this inference is almost always correct, so you typically don't need to specify `as_node`.

Specify `as_node` explicitly when the default inference doesn't apply:

- **Parallel branches**: If multiple nodes updated state in the same step, the checkpoint can't determine which was last, and LangGraph raises an `InvalidUpdateError`.
- **No execution history**: When setting up state on a fresh thread (common in [testing](/oss/langgraph/test)), there's no version history to infer from.
- **Skipping nodes**: Set `as_node` to a later node to make the graph think that node already ran and jump past it.

:::python
```python
# graph: generate_topic -> write_joke

# Treat this update as if generate_topic produced it.
# Execution resumes at write_joke (the successor of generate_topic).
fork_config = graph.update_state(
    before_joke.config,
    values={"topic": "chickens"},
    as_node="generate_topic",
)
```
:::

:::js
```typescript
// graph: generateTopic -> writeJoke

// Treat this update as if generateTopic produced it.
// Execution resumes at writeJoke (the successor of generateTopic).
const forkConfig = await graph.updateState(
  beforeJoke.config,
  { topic: "chickens" },
  { asNode: "generateTopic" },
);
```
:::

## Interrupts and time travel

If your graph uses @[`interrupt`] for [human-in-the-loop](/oss/langgraph/interrupts) workflows, replay and fork handle interrupts differently:

- **Replay** uses the cached resume value automatically — the interrupt is **not** re-triggered.
- **Fork** has no cached values — the interrupt **is** re-triggered and waits for a new `Command(resume=...)`.

In both cases, the node containing the interrupt **re-executes** — all node logic runs again. The difference is only in how `interrupt()` behaves: on replay it returns the cached answer silently, on fork it pauses and waits for new input.

<Tabs>
    <Tab title="Replay">
        Replaying through an interrupt node re-executes the node code, but `interrupt()` returns the cached resume value from the original execution. The user is not prompted again.

        :::python
        ```python
        from langgraph.types import interrupt, Command

        class State(TypedDict):
            value: list[str]

        def ask_human(state: State):
            answer = interrupt("What is your name?")
            return {"value": [f"Hello, {answer}!"]}

        def final_step(state: State):
            return {"value": ["Done"]}

        graph = (
            StateGraph(State)
            .add_node("ask_human", ask_human)
            .add_node("final_step", final_step)
            .add_edge(START, "ask_human")
            .add_edge("ask_human", "final_step")
            .compile(checkpointer=InMemorySaver())
        )

        config = {"configurable": {"thread_id": "1"}}

        # First run: hits interrupt
        graph.invoke({"value": []}, config)
        # Resume with answer
        graph.invoke(Command(resume="Alice"), config)

        # Replay from before ask_human
        history = list(graph.get_state_history(config))
        before_ask = [s for s in history if s.next == ("ask_human",)][-1]

        replay_result = graph.invoke(None, before_ask.config)
        # Interrupt NOT re-triggered — uses cached "Alice"
        # Result: {"value": ["Hello, Alice!", "Done"]}
        ```
        :::

        :::js
        ```typescript
        import { interrupt, Command } from "@langchain/langgraph";

        function askHuman(state: { value: string[] }) {
          const answer = interrupt("What is your name?");
          return { value: [`Hello, ${answer}!`] };
        }

        function finalStep(state: { value: string[] }) {
          return { value: ["Done"] };
        }

        // ... build graph with checkpointer ...

        // First run: hits interrupt
        await graph.invoke({ value: [] }, config);
        // Resume with answer
        await graph.invoke(new Command({ resume: "Alice" }), config);

        // Replay from before askHuman
        const states = [];
        for await (const state of graph.getStateHistory(config)) {
          states.push(state);
        }
        const beforeAsk = states.filter((s) => s.next.includes("askHuman")).pop();

        const replayResult = await graph.invoke(null, beforeAsk.config);
        // Interrupt NOT re-triggered — uses cached "Alice"
        ```
        :::
    </Tab>
    <Tab title="Fork">
        Forking before an interrupt node re-triggers the interrupt. The user must provide a new answer.

        :::python
        ```python
        # Using the same graph and completed execution from the Replay example

        # Fork from before ask_human
        history = list(graph.get_state_history(config))
        before_ask = [s for s in history if s.next == ("ask_human",)][-1]

        fork_config = graph.update_state(before_ask.config, {"value": ["forked"]})

        fork_result = graph.invoke(None, fork_config)
        # Interrupt IS re-triggered — no cached resume
        # Result contains __interrupt__, waiting for new Command(resume=...)

        # Resume with different answer
        graph.invoke(Command(resume="Bob"), fork_config)
        # Result: {"value": ["forked", "Hello, Bob!", "Done"]}
        ```
        :::

        :::js
        ```typescript
        // Using the same graph and completed execution from the Replay example

        // Fork from before askHuman
        const states = [];
        for await (const state of graph.getStateHistory(config)) {
          states.push(state);
        }
        const beforeAsk = states.filter((s) => s.next.includes("askHuman")).pop();

        const forkConfig = await graph.updateState(beforeAsk.config, { value: ["forked"] });
        const forkResult = await graph.invoke(null, forkConfig);
        // Interrupt IS re-triggered — waiting for new Command({ resume: ... })

        // Resume with different answer
        await graph.invoke(new Command({ resume: "Bob" }), forkConfig);
        // Result: { value: ["forked", "Hello, Bob!", "Done"] }
        ```
        :::
    </Tab>
</Tabs>

### Multiple interrupts

If your graph collects input at several points (for example, a multi-step form), you can fork from between the interrupts to change a later answer without re-asking earlier questions. Only interrupts **after** the fork point are re-triggered.

:::python
```python
def ask_name(state):
    name = interrupt("What is your name?")
    return {"value": [f"name:{name}"]}

def ask_age(state):
    age = interrupt("How old are you?")
    return {"value": [f"age:{age}"]}

# Graph: ask_name -> ask_age -> final
# After completing both interrupts:

# Fork from BETWEEN the two interrupts (after ask_name, before ask_age)
history = list(graph.get_state_history(config))
between = [s for s in history if s.next == ("ask_age",)][-1]

fork_config = graph.update_state(between.config, {"value": ["modified"]})
result = graph.invoke(None, fork_config)
# ask_name result preserved ("name:Alice")
# ask_age interrupt IS re-triggered — waiting for new answer
```
:::

:::js
```typescript
// Fork from BETWEEN the two interrupts (after askName, before askAge)
const states = [];
for await (const state of graph.getStateHistory(config)) {
  states.push(state);
}
const between = states.filter((s) => s.next.includes("askAge")).pop();

const forkConfig = await graph.updateState(between.config, { value: ["modified"] });
const result = await graph.invoke(null, forkConfig);
// askName result preserved ("name:Alice")
// askAge interrupt IS re-triggered — waiting for new answer
```
:::

## Subgraphs and time travel

If you fork a parent graph that contains a [subgraph](/oss/langgraph/use-subgraphs) with an @[`interrupt`], whether the subgraph interrupt re-triggers depends on the subgraph's [checkpointer configuration](/oss/langgraph/use-subgraphs#subgraph-persistence):

- **Default** (no `checkpointer` setting): The subgraph inherits the parent's checkpointer. Forking the parent **does** re-trigger subgraph interrupts.
- **`checkpointer=True`**: The subgraph has its own persistent checkpoints. Forking the parent does **not** re-trigger subgraph interrupts — you must fork from the subgraph's own checkpoint instead.

<Tabs>
    <Tab title="Default (stateless)">
        Forking the parent re-triggers the subgraph interrupt because the subgraph's state is stored in the parent's checkpoint.

        :::python
        ```python
        # Subgraph with default checkpointer (inherits parent)
        subgraph = (
            StateGraph(SubState)
            .add_node("ask_user", ask_user_node)  # Has interrupt()
            .add_edge(START, "ask_user")
            .compile()
        )

        # Parent graph
        graph = (
            StateGraph(ParentState)
            .add_node("router", router)
            .add_node("sub", subgraph)
            .add_edge(START, "router")
            .add_edge("router", "sub")
            .compile(checkpointer=InMemorySaver())
        )

        # Run, hit interrupt, resume
        graph.invoke({"value": []}, config)
        graph.invoke(Command(resume="answer"), config)

        # Fork from parent checkpoint before subgraph
        history = list(graph.get_state_history(config))
        before_sub = [s for s in history if s.next == ("sub",)][-1]
        fork_config = graph.update_state(before_sub.config, {"value": ["new"]})

        result = graph.invoke(None, fork_config)
        # Interrupt IS re-triggered
        ```
        :::

        :::js
        ```typescript
        // Subgraph with default checkpointer (inherits parent)
        const subgraph = new StateGraph(SubState)
          .addNode("askUser", askUserNode)  // Has interrupt()
          .addEdge(START, "askUser")
          .compile();

        // Parent graph
        const graph = new StateGraph(ParentState)
          .addNode("router", router)
          .addNode("sub", subgraph)
          .addEdge(START, "router")
          .addEdge("router", "sub")
          .compile({ checkpointer: new MemorySaver() });

        // Run, hit interrupt, resume
        await graph.invoke({ value: [] }, config);
        await graph.invoke(new Command({ resume: "answer" }), config);

        // Fork from parent checkpoint before subgraph
        const states = [];
        for await (const state of graph.getStateHistory(config)) {
          states.push(state);
        }
        const beforeSub = states.filter((s) => s.next.includes("sub")).pop();
        const forkConfig = await graph.updateState(beforeSub.config, { value: ["new"] });

        const result = await graph.invoke(null, forkConfig);
        // Interrupt IS re-triggered
        ```
        :::
    </Tab>
    <Tab title="checkpointer=True (stateful)">
        Forking the parent does **not** re-trigger the subgraph interrupt because the subgraph maintains its own checkpoint history. To re-trigger, fork from the subgraph's own checkpoint.

        :::python
        ```python
        # Subgraph with its own checkpointer
        subgraph = (
            StateGraph(SubState)
            .add_node("ask_user", ask_user_node)  # Has interrupt()
            .add_edge(START, "ask_user")
            .compile(checkpointer=True)
        )

        # Forking the parent does NOT re-trigger subgraph interrupt
        fork_config = graph.update_state(before_sub.config, {"value": ["new"]})
        result = graph.invoke(None, fork_config)
        # Interrupt NOT re-triggered

        # To re-trigger: fork from the subgraph's own checkpoint
        state = graph.get_state(config, subgraphs=True)
        sub_config = state.tasks[0].state.config
        sub_fork = graph.update_state(sub_config, {"value": ["sub_modified"]})
        result = graph.invoke(None, sub_fork)
        # Interrupt IS re-triggered
        ```
        :::

        :::js
        ```typescript
        // Subgraph with its own checkpointer
        const subgraph = new StateGraph(SubState)
          .addNode("askUser", askUserNode)  // Has interrupt()
          .addEdge(START, "askUser")
          .compile({ checkpointer: true });

        // Forking the parent does NOT re-trigger subgraph interrupt
        const forkConfig = await graph.updateState(beforeSub.config, { value: ["new"] });
        const result = await graph.invoke(null, forkConfig);
        // Interrupt NOT re-triggered

        // To re-trigger: fork from the subgraph's own checkpoint
        const state = await graph.getState(config, { subgraphs: true });
        const subConfig = state.tasks[0].state.config;
        const subFork = await graph.updateState(subConfig, { value: ["sub_modified"] });
        const subResult = await graph.invoke(null, subFork);
        // Interrupt IS re-triggered
        ```
        :::
    </Tab>

## Reference

### Common pitfalls

- **`update_state` is not rollback**: It creates a new checkpoint that branches from the specified point. The original history remains intact.
- **Replay re-executes nodes, not just reads from cache**: Nodes after the checkpoint run again. LLM calls fire again and may return different results. Only interrupt resume values are cached.
- **Replay uses cached resume values**: When replaying through an interrupt node, the original resume value is used automatically. Use fork if you need a new answer.
- **Subgraph checkpointer affects fork behavior**: With `checkpointer=True`, subgraph checkpoints persist independently of the parent. Fork from the subgraph's own checkpoint to re-trigger its interrupts.
- **Replay from the final checkpoint is a no-op**: If the graph has already completed (no `next` nodes), replaying produces the same result without re-executing anything.
